第 1 章整体心智模型

把系统想成 3 层。每一层有它的职责,层之间通过明确的"接口"对话。

三层各自的"边界"

层跑在哪儿关键文件能干的事
客户端 用户的浏览器 src/app/**/*.tsx
(标 'use client')		 表单状态、按钮交互、地图渲染。看不到数据库。
服务端 你的笔记本(NSSM 守着) src/app/**/*.tsx
src/app/**/_actions/*		 会话校验、业务逻辑、调用 Prisma、调用翻译 API。
数据层 项目根目录 dev.db
storage/		 SQLite 存结构化数据。storage/ 存文件(图片/PDF/视频)。
关键认知:客户端组件和服务端组件长得几乎一样(都是 .tsx),区别只在文件顶部有没有 'use client' 这一行。没有这行 = 服务器组件 = 能直接读数据库。有这行 = 客户端组件 = 只能用 hook、不能碰 prisma。

双语字段三件套

这是整个项目最有特色的设计。需要双语显示的字段不是一个字段,而是三个绑定的字段。

// 比如供应商名字
model Supplier {
  nameZh                 String     // 中文名(必填,Admin 录入)
  nameRu                 String?    // 俄文名(可空,AI 翻译 or Admin 手填)
  nameRuAutoTranslated   Boolean    // 标记:这条俄文是 AI 自动翻译的吗?
  ...
}
为什么要 _auto_translated 这个 boolean?

最初设计:Admin 手改俄文 → flag=false(锁定)→ 下次翻译跳过这个字段(避免覆盖人工修正)。2026.6.16 移除了这个"锁"机制:实际用着反而麻烦,改完后想重翻全部,得逐个清锁。现在每次翻译都覆盖现有俄文,简单一致。_auto_translated 字段仍在 schema 里(向后兼容,不删数据),只是 UI 不再感知,业务层永远写 true。

Supplier 表上有 11 组双语字段:nameZh+nameRu / shortName / province / city / district / address / description / mainProducts(2026.6.16 新增,"主营产品")/ originalIntentProductName / originalIntentOverview(2026.7.5 新增,"原始意图",列表页直读直显)。每组 3 个字段。

onDelete 4 种规则

外键删除时的连锁反应有 4 种,设计时不是凭感觉拍,是按"关系性质 + 字段可空性"两个维度决策的。

类型规则意思例子
附属Cascade主表删,附属也删Supplier 删 → 它的 Contact / Quote / Note 都删
审计Restrict有引用 = 禁止删主体User 的 createdBy —— 防止"删用户连带删走他所有数据"
共享Restrict资源被引用就不能删Tag —— 防止"删一个 Tag 影响 30 家挂着它的供应商"
弱关联SetNull主体删 → 引用字段置空Note.quoteId —— 关联的 Quote 没了,Note 本身还有价值,保留并把外键置空

第 4 章关键抽象层

所有"将来可能换"的能力都在 src/lib/ 下用抽象层封装,业务代码看不到底层。

🗄 数据库
业务代码用 prisma.supplier.findMany()。底层是 SQLite,将来切 PostgreSQL 改 schema.prisma 的 datasource 一行就行,业务代码 0 改动。
📁 文件存储
业务代码用 storage.put(key, buf)。底层现在写本地 storage/ 目录,将来切阿里 OSS 改一个 adapter 文件,业务代码 0 改动。
🌐 翻译服务
业务代码用 translateBatch([...])。底层现在 DeepSeek,可切 DeepL / Google,通过环境变量 TRANSLATE_PROVIDER 一行配置切换。
🔐 认证
业务代码用 requireCurrentUser()。底层是 better-auth,处理 session / cookie / 密码哈希。换库需要改业务调用点,但目前 better-auth 够用,不动。
🌏 i18n
业务代码用 const t = useTranslations('forms.contact')。底层是 next-intl,语言文件在 messages/。
👮 权限
业务代码用 isOwner(record, user)。规则:Admin 兜底所有人,EDITOR 只能动自己创建的。规则简单,没单独表。
抽象层的价值不是"现在",是"将来"。 现在切换需求几乎没有(本地跑得好好的),但 1 年后真上云时,代码改动量从"几百个文件"压缩到"几个 adapter 文件"。这是工程上的复利。

① Prisma Schema 里的 Supplier 表(数据模型的"宪法") 
model Supplier {
  id                     Int      @id @default(autoincrement())
  code                   String   @unique         // 人类可读编号 GZ-001

  // 双语三件套(共 11 组,这里只列 name)
  nameZh                 String                    // 中文必填
  nameRu                 String?                   // 俄文可空
  nameRuAutoTranslated   Boolean  @default(true)   // 翻译锁标记

  // 地理
  latitude               Float
  longitude              Float

  cooperationLevel       CooperationLevel @default(INITIAL_CONTACT)
  isActive               Boolean  @default(true)   // 软删除标记

  createdById            String
  createdBy              User     @relation(fields: [createdById], references: [id], onDelete: Restrict)

  // 反向关系
  contacts               Contact[]
  quotes                 Quote[]
  notes                  Note[]
  transactions           Transaction[]
  files                  File[]

读这段你能看出:① 双语三件套机制 ② 软删除模式(isActive 而不是物理删) ③ 反向关系(contacts: Contact[] 让你 supplier.contacts 直接拿)

② 权限 helper(src/lib/auth.ts) 
export async function requireCurrentUser(): Promise<CurrentUser> {
  const id = await requireUserId();
  const user = await prisma.user.findUnique({
    where: { id },
    select: { id: true, role: true },
  });
  if (!user) redirect('/login');  // session 引用了已删除的用户
  if (user.role !== 'ADMIN' && user.role !== 'EDITOR') {
    throw new Error(...);  // 运行时校验,SQLite 软枚举兜底
  }
  return { id: user.id, role: user.role };
}

export function isOwner(
  record: { createdById: string },
  user: CurrentUser,
): boolean {
  return user.role === 'ADMIN' || record.createdById === user.id;
}

整套权限模型就 8 行代码。 ADMIN 永远 true,EDITOR 看自己 createdById 是不是匹配。简单到不需要单独 Permission 表。

③ Server Action 的标准结构(updateContact) 
'use server';  // 文件顶部声明 = 整个文件是 Server Actions

export async function updateContact(
  contactId: number,
  _prevState: ContactFormState,
  formData: FormData,
): Promise<ContactFormState> {
  // ① 用 zod 校验输入
  const parsed = contactCreateSchema.safeParse(Object.fromEntries(formData));
  if (!parsed.success) return { status: 'error', errors: ... };

  // ② 查现有记录
  const existing = await prisma.contact.findUnique({ where: { id: contactId } });
  if (!existing) return { status: 'error', message: '联系人不存在' };

  // ③ 权限检查
  const user = await requireCurrentUser();
  if (!isOwner(existing, user)) {
    return { status: 'error', message: '只能修改自己创建的联系人' };
  }

  // ④ 实际写库
  await prisma.contact.update({ where: { id: contactId }, data: ... });

  // ⑤ 清缓存 + 跳转
  revalidatePath(`/suppliers/${existing.supplierId}`);
  redirect(`/suppliers/${existing.supplierId}`);
}

5 步是固定套路:校验 → 查 → 权限 → 写 → 跳。所有 update 类 server action 都是这个结构,你看一个就能改其他所有的。

④ 双语字段的 UI 渲染(BilingualFieldRow 抽象)

表单里"中文姓名 / 俄文姓名 + 已手改 badge"这种重复结构,封装成一个组件。看怎么实现:

function BilingualFieldRow({ pair, bi, onZhChange, onRuChange, manualEditLockedLabel }) {
  const isLocked = !bi[pair.flagFieldName];  // auto_translated=false 即"上锁"

  return (
    <div className="grid grid-cols-2 gap-4">
      <FormField label={pair.zhLabel}>
        <Input name={pair.zhFieldName} ... />       // 中文输入框
      </FormField>

      <FormField label={
        <>
          <span>{pair.ruLabel}</span>
          {isLocked && (
            <span className="text-warning">
              🔒 {manualEditLockedLabel}                  // "已手改" badge
            </span>
          )}
        </>
      }>
        <Input name={pair.ruFieldName} onChange={(e) => {
          onRuChange(...);   // 一改就翻 flag = false(锁定)
        }} />
        <input type="hidden" name={pair.flagFieldName} value={isLocked ? "false" : "true"} />
      </FormField>
    </div>
  );
}

核心 insight:"已手改"不是用户主动勾的,是用户编辑了俄文那一刻自动触发的。下次 Admin 点"自动翻译俄文",这个字段会被跳过。

⑤ Middleware 守门(src/middleware.ts) 
export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;

  // 公开路径直接放行
  if (PUBLIC_PATHS.some((p) => pathname.startsWith(p))) {
    return NextResponse.next();
  }

  // 检查 cookie 有没有 session
  const sessionCookie = getSessionCookie(request);
  if (!sessionCookie) {
    return NextResponse.redirect(new URL('/login', request.url));
  }

  return NextResponse.next();
}

每次浏览器请求 都会先经过这里。10 行代码守住所有未登录访问 —— 这就是"洋葱皮"模式的优雅之处。

⑥ i18n 切换语言的 Server Action(set-locale) 
'use server';

export async function setLocale(locale: 'zh' | 'ru') {
  const cookieStore = await cookies();
  cookieStore.set('NEXT_LOCALE', locale, {
    path: '/',
    maxAge: 60 * 60 * 24 * 365,  // 一年
  });
}

语言偏好存 cookie,服务器端读 cookie 决定加载 zh.json 还是 ru.json。状态在 cookie 里,刷新页面不会丢。

跟 AI 协作的提问范本

当你想让 AI 帮你做事时,带上"上下文 + 期望"两件套,避免它给你瞎写。

✅ 好的提问:
"我要给 Supplier 表加 foundedYear(Int? 可空)字段。
按 CLAUDE.md 里"加新字段"的 9 步流程帮我推进:
- 先改 schema.prisma

不要一次给所有代码,我们一步步来。"

关键点:

  • 明确字段类型(Int? 还是 Int?可空?默认值?)
  • 引用 CLAUDE.md 里的现有约定,避免 AI 自己发明规则
  • 限定"一步步来",不要一次性大泥球
  • 遇到不懂的术语,问"用普通话解释一下,不要堆术语"
